The System Design Manifesto

a sane path toward systems that stay small, change safely, and can be understood by one person

  1. Form Follows Function - the design is derived from the requirements, never the requirements from the design

    1. Requirements first. Bend the design to the requirements, not the requirements to a design you already like.

    2. The minimal change that meets them, with the least tech-debt accrued - change as little as possible while keeping future options broad.

    3. Future-proofing is a mechanism, not a wish - it is a decision already in this design that lets an anticipated change land later without rearchitecting. If you cannot point to that mechanism, it is not future-proofing; state it as a limitation and move on.

  2. A design is reviewable by one normal engineer

    1. If it takes a genius, or deep knowledge of the implementation, to follow your architecture, it is either too complex or too deep in the weeds. Neither is architecture.

    2. Present it one level up - the scope of the change and its effect on its neighbours (callers, downstream, sibling components), not a guided tour of the internals.

    3. Architecture is bound to change as understanding evolves. Capture the change and review it like you review code - the bigger the change, the more thorough the review.

  3. New is the Enemy

    1. New tools, services, and datastores are expensive - building is only ~25% of total cost of ownership1; the rest is the years of operation nobody budgets for.

    2. Reuse existing components, standards, and the ubiquitous language before inventing. Don't roll your own2 - a tool for what you need very likely already exists.

    3. When something genuinely new is required, build it reusably, so the next person does not reinvent it next quarter.

  4. Do One Thing, Well

    1. One component, one job. Every component does one thing and does it well (=a single responsibility, the fewest side-effects you can manage)3. This is the driving logic behind everything you design, from a single function to a whole service.

    2. Extend before you build (see New is the Enemy), but only while the component stays one thing. The test is the word and: the moment you need it to say what a component does (=it senses temperature and schedules jobs), it has become two components. Split it, don't grow it.

    3. All outputs are inputs. Design every output to be consumed by another program later (documented, parsable, uncluttered), even one that does not exist yet. Small pieces compose; monoliths do not.

    4. All outputs are inputs. Design every output to be consumed by another program later - documented, parsable, uncluttered. The component you feed may not exist yet.

  5. Less Logic, Fewer Bugs

    1. Code is hostile to your goals; deletion beats addition. Adding logic introduces new bugs; removing logic at most exposes existing ones, and reverts cleanly.

    2. Components talk through a well-defined, documented, versioned schema or API - agreed and tested by both sides.

    3. A datastore sits behind that API so it can be replaced with no disruption. Field names and output stay stable regardless of the system that transports them.

  6. Control at the Edge

    1. Put user-facing control as close to the user as possible - at the surface they already touch, not baked into the infrastructure.

    2. A lever buried in infra is one the user cannot reach and you cannot move without a migration.

    3. The cause a user acts on should live where the user is.

  7. Fail Correctly

    1. Every component will fail - that is fine. A cascading failure is not.

    2. Leave the kill switches, the max/min thresholds, and the noted margins of error, so a component fails in a known, predictable way and the rest of the system survives.

    3. Validate and type every input at the boundary, and alert on the invalid - input validation is the last line of defence against upstream bugs. You cannot fail because someone upstream did.

  8. Actionable by Design

    1. Every view, output, or metric ties an observed effect back to a cause the user can act on. A number with no lever is half a feature.

    2. Define health metrics and smells - a counter that should only rise, a rate that should never exceed its budget, a queue that should never grow unbounded.

    3. The user has the final say on value: an issue with no effect on the user might as well not exist; a behaviour the user perceives as wrong is wrong, even if implemented perfectly.

  9. Present Alternatives

    1. When more than one real option exists, present them with their pros and cons and name the flaws with pride - it is the only way to make an informed decision, and to grow as engineers.

    2. Don't smuggle a single solution past review; "there was only one way" is rarely true.

    3. When no option is clearly better, let the person who will build it choose - it secures ownership of the result.

  1. The Iceberg Fallacy: the cost of developing a new piece of software is only ~25% of the total cost of ownership that management sees and budgets for. The rest is operation, maintenance, integration, and the bugs you have not met yet. ↩︎

  2. Greenspun's Tenth Rule, generalized: any sufficiently complicated custom-built system contains an ad-hoc, informally-specified, bug-ridden, slow implementation of half of the industry standard it refused to adopt. The same is true of custom authentication, custom serialization, custom schedulers, and custom config languages. ↩︎

  3. The Unix philosophy, as Doug McIlroy set it down in 1978: "Make each program do one thing well. To do a new job, build afresh rather than complicate old programs by adding new features. Expect the output of every program to become the input to another, as yet unknown, program." It predates the microservice by decades and still decides the boundary. ↩︎
